npm的语义化版本

2022年12月12日 1.1k 字 大概 5 分钟

SemVer规范

​ 由 Gravatars 创办者兼 GitHub 共同创办者 Tom Preston-Werner 就建立了语义化版本控制的规范， semantic version 简称 semver, 于是这个规范就叫做 SemVer 规范，规范地址

版本号的格式

​ 标准的版本号必须采用X.Y.Z的格式,其中X,Y,Z为非负的整数,且禁止在数组前面补0.X是主版本号,Y是次版本号,Z是修订号, 英文表示为major, minor, patch, 每个元素必须以数值递增.

​ 主版本号为0的软件处于开发初始阶段,一切都可能随时发生改变.

​ 1.0.0的版本号用于界定公共API的形成

版本号递增逻辑

​ 修订号: 必须在只做了向下兼容的修正 (针对不正确的结果而进行的内部修改) 时才递增.

​ 次版本号:必须在有向下兼容的新功能出现时递增.在任何公共API的功能被标记为弃用时也必须递增.也可以在内部程序有大量新功能或改进被加入时递增,其中可以报考修订级别的改变.当次版本号改变时,修订号必须归零.

​ 主版本号: 必须在有任何不兼容的修改被加入公共API时递增.其中可以报考次版本号及修订号级别的改变.每当主版本号改变时,次版本号和修订号必须归零.

关于先行版本

​ 先行版本号可以被标注在修订版之后，先加上一个连接号再加上一连串以句点分隔的标识符来修饰。标识符必须由 ASCII 字母数字和连接号组成，且禁止留白。数字型的标识符禁止在前方补零，举个例子：1.0.0-alpha.6

​ 先行版的优先级低于相关联的标准版本，举个例子 1.0.0-alpha < 1.0.0

​ 被标上先行版本号则表示这个版本并非稳定而且可能无法满足预期的兼容性需求

alpha,beta和rc

​ alpha: 内部测试版,主要给开发和测试找bug

​ beta: 公开测试版,可以提前尝试一些新功能

​ rc: Release Candidate(候选版本)的缩写, 表示该版本的功能不再增加,和最终发布版的功能一样,有点像预览版.可能再找一找小bug,就会到正式版.

NPM指定版本范围

​ 在package.json文件中看到版本号前出现~ ^等字符,比如

  • ^ 表示次版本号的更新,比如^1.2.0表示安装的版本 >=1.2.0 < 2.0.0
  • ~ 表示修订号的更新, 比如~1.2.0表示安装的版本 >=1.2.0 < 1.3.0

​ 具体的逻辑:

  • ^ 只会执行不更改最左边非零数字的更新
  1. ^0.2.3 相当于 >=0.2.3 < 0.3.0
  2. ^0.0.3 相当于 >=0.0.3 <0.0.4
  3. ^1.2.3 相当于 >=1.2.3 < 2.0.0
  • ~ 如果指定了次版本号,则只会进行修订版本号的更新,如果没有指定,则会进行此版本号的更新
  1. ~1.2.3 相当于 >=1.2.3 < 1.3.0
  2. ~1.2 相当于 >=1.2.0 <1.3.0
  3. ~1 相当于 >=1.0.0 < 2.0.0

​ 其中最主要的就是 major、minor、patch，比如项目的 package.json 的 version 为 1.0.0：

  • 执行 npm version major 后，version 会变成 2.0.0
  • 执行 npm version minor后，version 会变成 1.1.0
  • 执行 npm version patch 后，version 会变成 1.0.1

​ premajor、preminor、prepatch 也是同理：

  • 执行 npm version premajor 后，version 会变成 2.0.0
  • 执行 npm version preminor后，version 会变成 1.1.0
  • 执行 npm version prepatch 后，version 会变成 1.0.1

​ 执行 npm version prerelease后，version 也会变成 1.0.1-0.

​ prerelease 和 prepatch的区别在于,prepatch 是增加 patch 号，先行版本号置为 0，prerelase 则是如果没有先行版本号，则跟 prepatch 一样，如果有的话，则会在先行版本号上加 1，举个例子，当版本号为 1.0.0-0时

​ 执行 npm version prepatch后，version 会变成 1.0.1-0

​ 而执行 npm version prerelease后，version 会变成 1.0.0-1
